  <files>

  <action>
- Create typed backup DTOs that separate file contract from raw DataStore keys. The envelope MUST contain:
  - schema version
  - app version name
  - export timestamp (UTC)
  - category manifest with stable IDs and sensitive-content flags
  - crypto header (algorithm, KDF, iterations, salt, IV)
  - encrypted payload bytes/text

- Lock the Phase 6 category list now so export/import UI can show it verbatim later. Use these stable category IDs:
  - `providers_models`
  - `provider_credentials`
  - `active_selections`
  - `language_defaults`
  - `prompts_profiles`
  - `app_mappings`
  - `transform_presets`
  - `keyboard_behavior`
  - `privacy_safety`
  - `advanced_preferences`

- Implement password encryption in `SettingsBackupCrypto.kt` using standard JCA only:
  - Derive a 256-bit key from the user password with `PBKDF2WithHmacSHA1` and random salt (use this algorithm specifically for API 24 compatibility; research uncertainty around SHA-256 support is not a reason to block the phase)
  - Encrypt/decrypt with `AES/GCM/NoPadding`
  - Persist KDF metadata in the envelope so future versions can migrate formats
  - Reject wrong-password/tampered payloads via normal AES-GCM auth failure; do not hand-roll checksum logic

- Add unit tests first in `SettingsBackupCryptoTest.kt`, then implement until they pass. Cover:
  - successful round trip
  - wrong password failure
  - tampered ciphertext failure
  - metadata/header values are populated and stable enough for future parsing
  </action>
